## Reflections

### On Test Coverage as Risk Management
Tests are the VaR of software engineering — they bound the tail risk of regressions. With 48 tests covering the core trading pipeline, we have reduced the probability of deploying a broken build. In a system that manages (paper) money, this is not optional.

### On the Monte Carlo Validation Post
The `signal-validation` repo demonstrates a gold standard for strategy validation:
1. **Out-of-sample only** (2016-2025)
2. **Random baseline** on identical dates
3. **Statistical significance** (p-value reported)
4. **Transparent methodology**

Our LLM trading system is more complex to validate because the "signal" is not a deterministic formula. But we can still:
- Compare LLM decisions vs random actions on same dates
- Compare LLM decisions vs a rules-based benchmark (e.g., RSI mean-reversion bot)
- Track win rate, Sharpe, and max drawdown over time

This should be a priority for the backtesting module.
